# Spectate Esports — Project Context

## What It Does
Spectate Esports is a pro Dota 2 match viewer. Users search for a team or tournament, see recent series results, and get a direct Twitch VOD link timestamped to the exact game. It also shows the full draft (picks, bans, player stats) and generates an AI match summary. The homepage shows live matches (with series scores and completed game chips), upcoming scheduled matches, and a Tournament Hub with standings, schedule, and format info.

Live at: https://spectateesports.live
GitHub: https://github.com/parthchoksi7/dota-match-finder

---

## Tech Stack
- **Frontend**: React + Vite + Tailwind CSS
- **Deployment**: Vercel (with Edge Middleware for OG tags + path-based routing)
- **Backend**: Vercel serverless functions (`/api/`)
- **Data**: OpenDota API (match data), Twitch API (VOD links), PandaScore API (live/upcoming/tournament data)
- **Cache**: Upstash Redis (KV) for live/upcoming/tournament caching; news articles cached 30min under `news:articles:dota2:v1`
- **News**: Multi-source aggregation via `api/news.js` — Steam Community RSS (official), Steam News JSON API (aggregates third-party feeds), Liquipedia MediaWiki API (roster/transfer news), and Currents API (editorial coverage); RSS fallbacks for PCGamesN and Dot Esports; no AI generation
- **AI**: Anthropic Claude Haiku via `/api/summarize.js`
- **Analytics**: Vercel Analytics (pageview/event tracking) + Vercel Speed Insights (Core Web Vitals / Real Experience Score) + Google Analytics (GA4) + BigQuery export
- **Analytics AI**: Claude Sonnet via `/api/analytics-chat.js` with BigQuery tool use for live GA4 data queries

- `src/api.js` - All API calls: OpenDota, Twitch VOD search, hero fetching, match summaries. `fetchHighlights(tournamentName)` fetches YouTube series highlights via `/api/tournaments?mode=highlights&name={name}`; results cached in module-level `_highlightsCache` Map by tournament name for the session. `matchHighlightsToSeries(videos, radiantTeam, direTeam, seriesStartTime)` is a pure client-side filter: keeps videos whose title contains either team name (case-insensitive substring), excludes videos published before `seriesStartTime` (Unix seconds), sorts ascending by `publishedAt`, returns the earliest post-match match — this correctly disambiguates two same-team series on the same day (the Group Stage clip at 4pm is excluded from the 7pm UB Final series because it predates the series start). Returns a full video object `{ videoId, title, publishedAt, thumbnail }` or null. `fetchMatchStats(matchId)` fetches end-game stats for a single match from `/api/tournaments?mode=match-stats&id={matchId}` — returns `{ radiantGoldAdv, players, events, itemNames, firstBloodTime, roshanKills }` or null on failure. `firstBloodTime` is seconds (integer, e.g. 83 → displayed as "1:23"); `roshanKills` is the count of Roshan kills derived from OpenDota `objectives[]`. Both are cached under KV key `stats:match:v8:{matchId}`. `events` is an array of `{ type: 'rapier'|'rampage'|'roshan', team: 'radiant'|'dire', player?: string, time: number, index?: number }` sorted by time ascending — `player` is set for rapier/rampage, `index` (1-based) is set for roshan. Module-level `_statsCache` Map prevents redundant fetches on drawer re-opens (same pattern as `_indicatorsCache`). `fetchTournamentPlayers(tournamentId, serieName, isCompleted)` fetches `/api/tournaments?mode=tournament-players&id={id}&name={name}` — returns `{ stats: { kills, deaths, assists, netWorth, gpm }, gameCount, league }` where each stat is a top-5 sorted array of `{ accountId, playerName, heroId, teamName, matchId, radiantName, direName, value, rank, gamesPlayed }`. Module-level `_tournamentPlayersCache` Map keyed by tournament ID. `fetchMatchIndicators(matchIds)` fetches game indicators (rapier/gold swing/mega comeback/rampage) from `/api/tournaments?mode=match-indicators&ids=...`; results are cached in a module-level `Map` for the browser session so repeated calls (series row mounts, card expands, drawer opens) hit memory instead of the network. `fetchProMatches()` fetches completed matches exclusively from OpenDota promatches. PandaScore is not used as a data source for completed matches — OpenDota is authoritative. Completed matches appear once OpenDota indexes them (typically 30min to several hours after a series ends). After the promatches fetch, a single `?mode=match-enrichment` call replaces the old two-call `match-formats` + `match-brackets` pattern — one KV round-trip returns both the series format corrections and bracket round labels for all matches on the page. **Pagination boundary guard**: handled entirely by `groupIntoSeries` in `HomeFeed.jsx` (operates on the full accumulated `allMatches` state, not per-page). The oldest incomplete series is dropped from the combined dataset; it reappears correctly once the user navigates to that date and the auto-load fetches the next page. There is deliberately NO per-page guard in `fetchProMatches` — a per-page guard caused a permanent data-loss bug where later games of a cross-page BO3 (e.g. games 2+3 on page N, game 1 on page N+1) were dropped from `allMatches` and never recovered. **Team name normalization**: `TEAM_NAME_MAP` constant maps OpenDota abbreviations to canonical names (e.g. `'BB' → 'BetBoom Team'`); applied when constructing match objects so display, stream fuzzy-matching, and follow logic all see the same string. **Twitch token prefetch**: module-level `getTwitchToken().catch(() => {})` fires at import time (fire-and-forget) so the token is warm before the user first opens a match drawer — eliminates the first-click waterfall where token fetch + VOD search were sequential. **Hero cache**: `fetchHeroes()` checks localStorage key `spectate-heroes-v1` first (24h TTL) before hitting OpenDota `/api/heroes`. Module-level `heroCache` variable serves as an L1 cache (session-lived); localStorage is L2 (survives page reloads). STORAGE_KEYS.HEROES = `"spectate-heroes-v1"`. **Hero search**: `resolveHeroByName(query)` returns `{ id, name, key }` for the first hero whose `localized_name` contains the query string (case-insensitive, min 2 chars), or null if not found; uses `fetchHeroes()` so it's always fast. `fetchHeroMatches(heroId, cursor?)` calls `?mode=hero-matches&hero_id={id}[&cursor={ts}]`; returns `{ rows, exhausted, cursor }`. `handleSearch` in `App.jsx` is `async`; it calls `resolveHeroByName` before the team/tournament search — if a hero is found, it navigates to `/heroes/{hero.key}` instead of running the search overlay.

- `src/components/GoldGraph.jsx` - Custom SVG gold advantage chart. Props: `radiantGoldAdv: number[]` (per-minute array from OpenDota), `radiantName: string`, `direName: string`, `loading?: boolean`, `events?: Event[]`, `vodUrl?: string`. Uses two `<path>` elements with `<clipPath>` to render green fill above the zero line (radiant ahead) and red fill below (dire ahead) — no chart library. `computePoints(data)` is exported for unit tests; it maps the array to SVG `{x, y}` coordinates using viewBox constants (VW=480, VH=160, PL=4, PR=4, PT=10, PB=22, CW=472, CH=128, MID=74). PL/PR are minimal stroke-buffer only — no labels inside the SVG. **Layout**: GoldGraph returns a React fragment: (1) an HTML header row (`flex justify-between px-5 mb-1.5`) showing RADIANT label (green), current gold diff in advantage color, and DIRE label (red); (2) the SVG wrapper div. In MatchDrawer, GoldGraph is wrapped in `-mx-5` so the SVG is full-bleed to the drawer panel edges; the header row's `px-5` realigns its text with other drawer content. **Scrubbing interaction**: hover anywhere on the chart (desktop) or drag horizontally (mobile) to see the gold advantage at any minute. Desktop: `onMouseMove` on the `<svg>` element → dashed vertical crosshair line + colored dot on the gold line + floating tooltip near cursor showing `"23m · +8.3k RADIANT"`; `onMouseLeave` dismisses. **Desktop hover tooltip** uses `position: fixed` anchored to `hoverViewport` state (clientX/clientY from mousemove) — `left: Math.max(8, Math.min(window.innerWidth - 210, clientX - 80))` — escapes drawer's `overflow-x-hidden` so it never clips at screen edges. Mobile: imperative `addEventListener('touchmove', ..., { passive: false })` with direction-intent detection (5px threshold) — horizontal drag captures input and calls `preventDefault()` to block drawer scroll; vertical swipe falls through to normal scroll; tooltip renders as the same `position: fixed` floating card as the desktop tooltip (same markup, same dark card style), positioned 70px above the touch point (`top: clientY - 70`) so it clears both the thumb and lollipop markers; `hoverViewport` state is set to `{ x: clientX, y: clientY }` during touchmove and cleared (along with `hoverMinute`) after the 600ms linger on `touchend`. `hoverSourceRef` ('mouse'|'touch') determines which tooltip variant renders. GA events: `gold_chart_scrub` (source: 'mouse'|'touch') fires once per scrub session via `hasTrackedScrubRef`; `gold_graph_marker_click` ({type, team}) fires on VOD-jump click. **Interactive event markers (lollipop style)**: four types — `roshan` (Aegis shield `RoshanSvg`), `rampage` (skull `RampageSvg`), `rapier` (sword `RapierSvg`), `teamfight` (crossed swords `TeamFightSvg`, cyan `#06b6d4`) — all from `GameIndicators.jsx`. Teamfight markers are produced server-side by `filterNotableTeamFights()` in `api/tournaments.js`: filters to ≥3 deaths and ≥500 net gold floor, then ranks qualifying fights by composite score `(deaths/maxDeaths)*0.4 + (|netGold|/maxNetGold)*0.6`, taking top 8. `team` field = winner (positive net → radiant, negative → dire). Tooltip shows winning team name, deaths count, net gold delta, minute, and WATCH link. Rendered by `GraphMarker` component. **Two-layer color system (Option F)**: icon + disc fill tint encodes event type (amber for Roshan, orange for Rampage, red for Rapier); outer ring encodes side (green `#22c55e` for Radiant, red `#ef4444` for Dire) — these are independent, so a Dire Roshan has an amber icon + red ring. **Lollipop anatomy**: thin stem from the data point, dot at the data point, outer ring (side color), filled disc (event-type tint), SVG icon. Radiant markers: stem rises UP, disc floats above the line. Dire markers: stem falls DOWN, disc hangs below. Active state: DISC_R=11, full opacity, ring 2.5px/100%; inactive: DISC_R=10, 65% disc/icon opacity, ring 2px/70%. Ring radius equals DISC_R (flush border, no halo gap); ring renders after disc fill so it acts as a colored border. `activeEvent` payload stores `discY` (disc center in SVG coords, always using active DISC_R=11 since the marker is active when the payload is used) and `chipColor` (event-type icon hue). Z-ordering: inactive markers rendered first, active last. Vertical dashed ruler at active marker's x. **Event marker tooltip** (inline-styled dark card): chip-colored icon (NOT side color) · event label · subject · minute · "WATCH" in amber if VOD available. `position: fixed` is mandatory — `position: absolute` is clipped by the drawer's `overflow-x-hidden` scroll container (MatchDrawer line 232) and its `overflow-hidden` panel. A `useLayoutEffect` on `activeEvent` reads `svgRef.current.getBoundingClientRect()` to convert SVG coords (using `activeEvent.discY`) to viewport pixels, reads `tooltipRef.current.offsetWidth` for the rendered width, then clamps `left` to `[8px, window.innerWidth - tipWidth - 8px]`. Tooltip starts at `left: -9999` (invisible) until the effect runs synchronously before paint — no visible snap. Flip direction (`(activeEvent.x - PL) / CW > 0.45`) determines which side of the marker to anchor. **Interaction model**: markers use `onMouseEnter`/`onMouseLeave` (mouse-only) — mobile is two-tap via `onClick`: first tap shows tooltip, second opens VOD. The `<svg>` element has `onClick={() => setActiveEvent(null)}` for tap-away dismiss. Click calls `buildEventUrl(vodUrl, event.time)` (parses `?t=` Twitch offset, adds event seconds) and opens new tab. Loading state: fragment with spacer + 160px `animate-pulse` skeleton. Empty/unavailable state (fewer than 2 data points): 160px div — "Gold data unavailable". Time axis labels every 5 minutes. Spoiler-free and PandaScore guards are applied by the parent (MatchDrawer).

### Backend (Vercel Serverless)
- `api/summarize.js` - Generates AI match summaries (Claude Haiku) and AI tournament summaries. Match mode: default POST with match data. Tournament mode: POST with `{ type: 'tournament', seriesId, name, leagueName, status, beginAt, endAt, prizePool, teams, stages }`; cached 24h (30 days for completed) under `tournament:summary:{id}`.
- `api/live-matches.js` - Fetches live Dota 2 matches from PandaScore; cached in KV for 2 min. `getSeriesScore(m)` reads raw per-team win counts from `m.results[].score`; `winsRequired(matchType, numberOfGames)` caps each score at the series maximum (BO3→2, BO5→3, etc.) before returning the string — without this cap PandaScore reports all games played (e.g. 3-0 for a BO3 sweep) instead of the series wins needed (2-0).

## PS ↔ OD Data Connection

This is the core mechanism that ties PandaScore's live/tournament data to OpenDota's match data (drafts, gold graphs, stats, VODs). Understanding it is essential before touching any live match, stream, or VOD logic.

### The only true linking field: `external_identifier`

Each PandaScore **game** object exposes `game.external_identifier`, which **is** the OpenDota match ID. It is mapped in `api/live-matches.js` `mapGames()`:

```js
matchId: g.external_identifier || null
```

**Critical constraint**: PandaScore only populates this field while the game is **running** (live). Once the game finishes, PandaScore clears it. Any logic that relies on this field must account for the null-after-finish behaviour.

### How the connection is persisted: KV cache during live play

Because `external_identifier` disappears after a game ends, `cacheRunningStreams()` in `api/live-matches.js` writes three KV entries while a game is live:

| KV key | Value | Written when | TTL |
|---|---|---|---|
| `live:game:{psMatchId}:{position}` | OD match ID | `game.status === 'running'` | 14 days |
| `stream:match:{odMatchId}` | Twitch channel handle | `game.status === 'running'`, `nx:true` (write-once) | 14 days |
| `format:match:{odMatchId}` | PS match_type (e.g. `"best_of_2"`) | any game in the match | 14 days |

The `live:game:` key is what allows finished games to recover their OD match ID: the normal handler does a batch `mget` of these keys and merges the IDs back into the `games` array before caching the response. The `format:match:` key allows the completed-match feed to correct OpenDota's `series_type` when it disagrees with PandaScore (e.g. DreamLeague group stage BO2s reported by OD as series_type 1 = BO3).

This caching is written by **both** the normal client-poll handler and the 30-min GitHub Actions cron (`?cron=1`). The cron runs even when no user is watching, ensuring coverage for games that start between user polls.

### Series-level: no direct ID mapping

There is **no PandaScore `serie_id` → OpenDota `series_id` mapping**. The two IDs come from completely separate systems. A series connection is derived by aggregating game-level links — once Game 1 and Game 2 each have an OD match ID, they share the same OpenDota `series_id`. This is implicit, not stored.

### Fallback: time + team name fuzzy matching

When no cached `external_identifier` exists (historical matches, qualifier series, cold KV), two fallbacks resolve the OD match:

1. **`findOdMatchByTime()`** in `api/_shared.js`: given a PS game's `begin_at` (Unix seconds) and its opponents array, filters OD promatches to a **±15 min** timestamp window (900s — PS `begin_at` is scheduled time; OD `start_time` is actual in-engine start after drafting, empirically 7–10 min later), then uses bidirectional substring team name matching to break ties. Used by `?mode=recent-completed`.

2. **`match-streams.js` PandaScore fuzzy match**: given an OD match's `start_time` and team names (`radiantTeam`/`direTeam` query params), queries PandaScore `/dota2/matches` with a **±1h** time window and runs `teamsMatch()`. If a match is found, its Twitch stream is extracted via `getTwitchStreams()` and cached to `stream:match:{odMatchId}` for future hits.

Both run through the same canonical helpers in `src/teamMatching.js` (2026-07-15: extracted out of `api/_shared.js` into a zero-import module — same pattern as `src/seriesLogic.js` — so the client can use them too, for favorites highlighting; `_shared.js` imports and re-exports `normalizeTeamName`/`teamPairMatch`/`teamPairScore` for its existing server call sites): `normalizeTeamName(name)` (lowercase + strip every non-alphanumeric char, Unicode-aware) feeding `teamPairMatch(psA, psB, odR, odD)` (order-independent **bidirectional substring** on the normalized names):
```js
x.includes(y) || y.includes(x)   // on normalizeTeamName(x), normalizeTeamName(y)
```
Normalization handles cosmetic separator differences (e.g. OD `"ggboom"` vs PS `"GG Boom"`, `"Virtus.pro"` vs `"Virtuspro"`); the bidirectional substring handles truncation/abbreviation on either side (e.g. `"BetBoom Team"` vs `"BetBoom"`, `"Yakult Brothers"` vs `"Yakult S Brothers"`). `teamPairMatch` returns false if any name normalizes to `''`, so a missing/TBD name never matches all. **Never replace this with exact equality or one-directional contains.** Any new PS↔OD name matching must use the same helpers, not a fresh copy.

**`findBestPsMatch(psMatches, odNameA, odNameB)` (2026-07-07)** — the canonical way to pick a PS match out of a list of same-time-window candidates; replaces the old per-caller `.find(m => teamsMatch(m.opponents, a, b))` pattern in `match-streams.js`, `enrich-stream-history.mjs`, and `backfill-streams-json.mjs` (all three now call this instead). Two passes: (1) strict `teamPairMatch` across all candidates — unchanged, zero-ambiguity behavior; (2) **score fallback**, only when no candidate strict-matches — `teamPairScore(psA, psB, odA, odB)` counts how many of the two OD names each candidate partially matches (0–2, one-sided substring instead of requiring both), and `findBestPsMatch` returns the candidate with a UNIQUE highest score ≥ 1. Resolves PandaScore and OpenDota using a genuinely different name for the same org on one side of a matchup — confirmed regression cases from EWC 2026's 2026-07-07 group stage: PS `"PlayTime"` vs OD `"PTime"`, PS `"Poor Rangers"` vs OD `"_PowerRangers"` — **without a hardcoded alias**. Important nuance: it doesn't "bridge" the divergent side itself (`"PlayTime"`/`"PTime"` never match each other, contributing 0 either way) — it works because the OTHER, unaffected side of the pairing (`"Team Liquid"`, `"GamerLegion"`) matches exactly and no other candidate in the window shares that name too, so score 1 is still uniquely the highest. If BOTH sides of a pairing diverge simultaneously, this can't help (score 0 for every candidate) — that needs a `TEAM_NAME_ALIAS_GROUPS` entry. Also returns `null` rather than guessing on a genuine tie between two equally-plausible partial matches. `TEAM_NAME_ALIAS_GROUPS` in `src/teamMatching.js` is the documented last-resort layer for a case the score fallback can't disambiguate on its own (a genuine tie, or both sides of a pairing diverging at once) — each entry is a group of `normalizeTeamName()` outputs known to refer to the same org, checked **additively** by `namesEquivalent()` alongside substring matching. This does NOT rewrite `normalizeTeamName`'s output — an earlier version did, which broke `"BetBoom Team"` matching any OD row that legitimately calls them `"BetBoom"` (caught by its own regression test before shipping). Seeded 2026-07-07 from a scrub of all tier-1 teams' PandaScore vs OpenDota team registries: `['betboomteam', 'boomboys']` — OpenDota's persistent team registry still carries the legacy name "BoomBoys" (team_id 8255888, 667 recorded wins) for the org PandaScore calls "BetBoom Team"; PandaScore's own team search returns zero hits for "BoomBoys", confirming it's a one-sided legacy name, not a different team. The same scrub found no other tier-1 gaps beyond what the existing truncation substring already handles (e.g. PS "Aurora" / OD "Aurora Gaming", PS "nouns" / OD "Nouns Esports") — `"OG"`'s PandaScore team id remains inconclusive (their 2-char name makes PS's search return noise; needs spot-checking next time they're in an active tournament). **`['1win', 'tundraesports']` added 2026-07-15**, confirmed live: PandaScore's opponent name for the July 2026 EWC Round 2 match (id 1565904) is `"1win"`, but OpenDota's team registry has no `"1win"` team_id at all — team_id 8291895 (the org that absorbed Tundra Esports' roster in June 2026) still carries `radiant_name`/`dire_name` `"Tundra Esports"` on its most recent match (OD match 8815912139, ~2026-05), i.e. OD's per-match naming hasn't caught up to the roster/branding change. **Favorites ("My Teams") highlighting (2026-07-15 fix)** — `isTeamFollowed(followedTeams, ...teamNames)` in `src/teamMatching.js`, re-exported via `src/utils.js`, is now the single comparison used everywhere a match/series needs to render as "followed" (`HomeFeed.jsx`'s My Teams card, live/upcoming/completed row highlighting; `buildTournamentCards`'s sort-to-top). Previously these were raw `followedTeams.includes(...)` exact-string checks, which broke whenever the team was followed via one provider's name (the follow star only appears on played matches, so `followedTeams` always holds an OpenDota-sourced name) but a match displayed the other provider's name for the same org — upcoming matches in particular are PandaScore-only (no OD cross-reference exists yet for a future fixture), so a team followed as OD's `"Tundra Esports"` never matched its own PandaScore-sourced `"1win"` upcoming fixture. The follow-star's own filled/toggle state (`MatchCard.jsx`, `MatchDrawer.jsx`) intentionally still uses exact matching — see the doc comment on `isTeamFollowed` for why fuzzy-matching the toggle target itself would be wrong (desyncs add/remove from the fuzzy-matched display, piling up near-duplicate `followedTeams` entries). `isTeamFollowed` deliberately does **not** reuse `namesEquivalent`/`teamPairMatch`'s bidirectional-substring rule as-is: that rule is only safe inside a two-sided pairing check (a false positive needs two independent coincidences in the same time window), but `isTeamFollowed` checks one followed name against one arbitrary candidate with no such cross-validation — caught in review as a false-positive regression where following "OG" (one of this app's own curated tier-1 teams) highlighted any match involving "Zero Gaming", "Turbo Gaming", or "Dogs" (all coincidentally contain "og" as a substring). Fixed with a local `followedNameEquivalent()` that requires exact-normalized-equality or an alias-group hit for short names, only allowing substring truncation (e.g. PS "Aurora" / OD "Aurora Gaming") when both sides normalize to ≥4 characters. `namesEquivalent`/`teamPairMatch`/`teamPairScore` themselves were left untouched to avoid any risk to the locked VOD system (`findBestPsMatch` depends on them).

### Live-game OD id capture — `live_game_map` (2026-07-16, Phase 0a)

A **second, independent** source for a game's OpenDota `match_id`, complementing PandaScore's `external_identifier`. `api/tournaments.js?mode=od-live-capture` (handler `api/_handlers/liveOdCapture.js`) snapshots OpenDota `/api/live` tier-1 games (league_id > 0, both team names, real match_id) into the Supabase `live_game_map` table (`scripts/create-live-game-map.sql`). OD `/live` exposes `match_id` **and** `series_id` **while the game is running** — before `/promatches` indexes the finished game (30–90 min lag) and after PandaScore clears `external_identifier` at game end — so this closes the window where the KV `live:game:` capture was missed.

- **Rows are OpenDota-keyed** (`od_match_id` unique): `od_series_id`, `radiant_name`/`dire_name`, `start_time` (= `/live activate_time`), `league_id`, plus nullable live telemetry (`radiant_lead`, `radiant_score`, `dire_score`, `server_steam_id`, `game_time`) for a future live-pulse feature. It does **not** store PandaScore ids — correlation to a PS game happens at resolve time via `findOdMatchByTime()` (rows are shaped to be a drop-in `odMatches` argument). This keeps the capture free of any PandaScore fetch and fully isolated from the LOCKED VOD stream cache (it never writes `live:game:`/`stream:match:`).
- **Trigger + throttle:** primarily the client's existing 2-min live poll (`fetchLiveData` in `App.jsx` fires `?mode=od-live-capture`, fire-and-forget, only when a game is `status === 'running'` — 0 QStash cost); a `*/15` QStash backstop (`setup-qstash-schedules.mjs`) covers no-user windows. A KV lock (`capture:od-live:lock`, `nx:true`, 110s TTL, never released) collapses all callers to ~one OD `/live` fetch per 2 min — this is the cadence control and the abuse guard, so the endpoint is intentionally unauthenticated. Upsert on `od_match_id` refreshes telemetry + `captured_at` each run.
- **Resolver (`?mode=live-series-games&id={psMatchId}`, `api/_handlers/liveSeriesGames.js`):** given a PandaScore series match id, returns `{ games: [{ position, matchId }], gameIds }` for its finished games. Resolution chain, most-authoritative first: (1) PandaScore `external_identifier` from the per-match endpoint; (2) KV `live:game:{psId}:{pos}`; (3) `match_stream_history` by exact `(ps_match_id, game_position)`; (4) `live_game_map` — team-name + `begin_at` fuzzy match via `findOdMatchByTime()`. **KV is backfilled only for authoritative hits (1/2/3); a fuzzy `live_game_map` correlation is returned but never written to `live:game:`**, so a wrong guess can't leak into the LOCKED live-matches.js enrichment. OpenDota `/promatches` is intentionally not queried here — once OD indexes a finished game it flows into the client's `allMatches` anyway. Degrades to a KV-only read of positions 1–5 when PandaScore is unavailable. The PandaScore fetch itself (own try/catch + 4s `AbortController`) is factored into `fetchPsMatchDetail(pandaId, log)`, exported for reuse by the Phase 2 live-pulse handler below.
- **Live pulse (`?mode=live-game-pulse&id={psMatchId}`, `api/_handlers/liveGamePulse.js`, 2026-07-16, Phase 2):** gold lead / kill score / live draft for the **currently RUNNING** game of a series (as opposed to the resolver above, which only covers finished games). `live_game_map` gained two columns for this — `radiant_hero_ids`/`dire_hero_ids` (integer arrays, split from OD `/live` `players[].hero_id` by `players[].team`: `0`=Radiant, `1`=Dire, confirmed empirically 2026-07-16 — every live league game splits exactly 5/5 across these two values). The handler resolves the running game the same fuzzy way as source (4) above (team-name + `begin_at` window via `findOdMatchByTime`) and returns the matched row's telemetry: `{ matchId, radiantName, direName, radiantLead, radiantScore, direScore, gameTime, radiantHeroIds, direHeroIds, capturedAt }` or `{ pulse: null }`. **Read-only — this correlation is never written anywhere** (not authoritative, purely for display). Frontend: `SeriesLivePulse.jsx` self-polls this mode every 20s while mounted (the sheet itself doesn't otherwise refresh faster than the app's 2-min live poll). Owner-gated (`spectate-owner`) alongside the rest of the companion UI as of this writing.

### Full resolution flow (drawer open)

```
User opens match drawer (OD match ID known)
  ↓
fetchMatchStreams(matchIds, ts, radiantTeam, direTeam)  [src/api.js]
  ↓
api/match-streams.js checks in order:
  1. KV  stream:match:{id}        — fast path (written during live play or prior fuzzy match)
  2. PandaScore fuzzy match        — ±1h time window + bidirectional team name match
     → on hit: cache result to KV, return channel
  3. ts bucket  stream:ts:{bucket} — candidate channels list (returned as _candidates;
     no longer used for VOD search — kept only for diagnostics)
```

The frontend (`findTwitchVod` in `src/api.js`) uses only `preferredChannel` from step 1 or 2 — it does **not** fall back to other channels. This prevents returning a VOD from a concurrent match on the same channel.

## Core Features

### Match Discovery
- Fetches pro matches from OpenDota `/promatches` endpoint
- Filter is a pure OR of two independent rules — a match is kept if EITHER passes:
  1. **OpenDota premium**: `fetchPremiumLeagueIds()` calls `/api/tournaments?mode=premium-league-ids` (server-side proxy — avoids CORS) which fetches `/api/leagues` and returns premium-tier league IDs. `premiumIds.has(m.leagueid)` returns true for premium matches. (professional tier is intentionally excluded; tier A events are covered via rule 2)
  2. **PandaScore tier S/A name match**: `fetchTier1LeagueNames()` fetches `api/tournaments?mode=tier1-leagues` — returns tier S/A league names (e.g. "DreamLeague", "ESL One", "PGL", "PGL Wallachia") from PandaScore merged with the hardcoded `PERMANENT_TIER1_NAMES` list. `matchesTier1Names(m.league_name, tier1Names)` does case-insensitive substring match. Names < 3 chars are skipped to avoid accidental broad matches (3-char names like "PGL" are allowed as generic org catch-alls). Both fetches run in parallel with the promatches request and are cached per session.
- Groups individual games into series (BO1/BO2/BO3/BO5) — OpenDota series_type 3 = BO2 (undocumented); BO2 draws (1-1) are also explicitly marked complete
- Search filters `allMatches` live so load more updates results automatically

### SEO Match URLs & Sitemap
- Match URLs use keyword-rich slugs: `/match/team-spirit-vs-gaimin-gladiators-dreamleague-s23-{matchId}`
- `slugify()` and `getMatchSlug()` in `App.jsx` generate the slug from team names, tournament, and match ID
- Match ID always at the end of the slug for reliable extraction: `pathname.match(/^\/match\/.*?(\d+)\/?$/)`
- Old hash URLs (`#match-{id}`) and numeric URLs (`/match/{id}`) still work - backwards-compatible
- `middleware.js` injects per-match OG meta tags (title, description, og:image) for social sharing and SEO
- **Match page titles (July 2026 SEO rewrite)**: `{Radiant} vs {Dire} — {tournament} | VOD, Draft & Stats` — intent keywords, no kill score, no winner spoiler in the title. Tournament display name prefers the Supabase `match_stream_history.tournament` (PandaScore name) over the OpenDota league name. The description names the winner, kill score, tournament, and date (LLM-extractable result statement). The h1 is `{Radiant} vs {Dire} — {tournament}` (no brand suffix).
- **Non-Tier-1 noindex gate (July 2026)**: match pages get `<meta name="robots" content="noindex, follow">` unless OpenDota `league.tier === 'premium'` OR the league/tournament name matches `TIER1_NAME_PATTERNS_SSR` (inline mirror of `PERMANENT_TIER1_NAMES` in `api/_shared.js` — keep in sync). Prevents regional/amateur leagues (e.g. "Dota 2 Space League") from diluting search quality. Applied only when OpenDota data resolved (no noindex on transient API failures).
- `api/sitemap.js` generates a full XML sitemap with slug URLs for recent matches. `match_stream_history` rows are gated by `isTier1TournamentName()` (name match against `PERMANENT_TIER1_NAMES`); rows that fail the gate can be rescued by the OpenDota premium fallback. Null-tournament rows are excluded.
- **Tournament URLs (July 2026)**: `/tournament/{keyword-slug}-{id}` (e.g. `/tournament/esports-world-cup-2026-10728`). ID always trailing, extracted with `pathname.match(/^\/tournament\/.*?(\d+)\/?$/)`. Middleware 301s bare-numeric and stale-slug URLs to the canonical slug. Slug builders (keep in sync): `tournamentPathMw()` in `middleware.js`, `tournamentUrlFromSeries()` in `api/sitemap.js`, `tournamentPath()` in `src/utils.js`. All build from `seoName` (see Tournament Hub Pages below).
- **Soft 404 handling**: when a `/match/` URL fails to resolve (not in live feed, and OpenDota returns no data), `App.jsx` injects `<meta name="robots" content="noindex, nofollow">` so crawlers drop the stale URL. `TournamentDetail.jsx` does the same when the tournament-detail API returns 404 or any error.
- **IndexNow**: `pingIndexNow()` in `api/_shared.js` (requires `INDEXNOW_KEY` env; key file in `public/`). Pinged from: sitemap generation (recent <48h matches + live tournament hubs) and `publishToDb()` in `api/pipeline/_publisher.js` (new articles). Bing's index powers ChatGPT search retrieval.

### Live Matches (PandaScore)
- `api/live-matches.js` calls PandaScore `/dota2/matches/running`
- Filters to tier-S and tier-A tournaments using `isTier1(m) || isTier1ByName(m, names)`. Primary: checks `match.tournament.tier`. Fallback: checks `match.league.name` against a merged names array: KV-cached tier1 names plus `PERMANENT_TIER1_NAMES` (hardcoded in `_shared.js`). The hardcoded list ensures DreamLeague, ESL One, PGL, etc. always pass even when KV is cold (e.g. fresh Redis flush). This matters for qualifier matches where PandaScore sets `tournament.tier = "c"` despite being a tier1 organizer event. Maps each match to `{id, teamA, teamB, tournament, seriesLabel, bracketRound, seriesScore, currentGame, games, streams}`
- `bracketRound` - extracted from `m.name` by stripping everything after the first `:` and applying title case (e.g. "Upper Bracket Final: PARI vs TS" → "Upper Bracket Final"). Shown in `LiveMatchRow` below the score row. KV key: `dota2:live_matches_v4`
- `seriesScore` - derived from `m.results` (per-team win counts mapped by team ID), capped by `winsRequired(m.match_type, m.number_of_games)` to prevent showing raw game totals (e.g. BO3 score is capped at 2; PandaScore can report score=3 for a BO3 sweep)
- `currentGame` - position of the game with `status === 'running'`
- `games` - array of `{position, status, winnerName, matchId, beginAt, length}` where `matchId` is `external_identifier` (OpenDota match ID). For finished games where `external_identifier` is null (only populated while a game is running), the handler does a batch `mget` of `live:game:{psMatchId}:{position}` KV keys and merges the resolved OD IDs back into the game objects.
- `youtubeStream` - first English YouTube URL from `streams_list` (rendering-layer only; `getTwitchStreams()` in `_shared.js` is not modified)
- Cached in Upstash Redis for 2 minutes; bust cache via `?bust=1`
- **Mid-series click behavior (REQ-2)**: clicking a live match row with at least one finished game opens `LiveSeriesSheet` (not `MatchDrawer` directly). `handleSelectLiveMatch` in App.jsx reads `liveMatches` state, finds the PS match, and calls `setSelectedLiveSeries(match)`. The replay button in `LiveSeriesSheet` calls `handleLiveSeriesReplay(odMatchId)` which closes the sheet then opens the full OD `MatchDrawer` via `handleSelectMatchId`. **Data routing (REQ-3)**: PS data is used ONLY while `match.status === 'running'`; once PS marks a series `finished` it no longer appears in `/dota2/matches/running` and OD is the sole data source.
- Spoiler-free mode: hides series score behind the intentional `?·?` curtain marker when a score exists (`vs` is shown only when there is genuinely no series score yet), hides winner names in chips and LiveSeriesSheet rows, disables team dimming

### Upcoming Matches (PandaScore)

Maps each match to `{id, scheduledAt, teamA, teamB, tournament, seriesLabel, bracketRound, streams}`. `bracketRound` strips after `:` from `m.name` and applies title case; shown in `UpcomingMatchRow` below teams, above time. ### VOD Linking
- PandaScore is the authoritative source for which Twitch channel streamed a match
- **Stream mapping (server-side cron)**: `api/live-matches.js?cron=1` is triggered every 15 min by a **QStash schedule** (provisioned via `npm run setup-qstash`; see `scripts/setup-qstash-schedules.mjs`). Moved off GitHub Actions on 2026-06-26 — GHA throttling fired the declared `*/15` cron only every ~1.5-4h in practice. `.github/workflows/stream-capture.yml` remains as a manual-dispatch fallback only; `auto-tweet.yml` still pings `cron=1` every 30 min as backup. It fetches up to 100 running matches from PandaScore (`page[size]=100`, PandaScore's max) and writes `stream:match:{gameMatchId}` (nx:true - write-once) and `stream:ts:{roundedBeginAt}` (JSON array of active channels) to KV (14-day TTL). A backup trigger also runs from `auto-tweet.yml` every 30 min. Cron runs are authenticated via `CRON_SECRET` header. **Cadence matters:** a Dota game runs ~35 min, so the previous 30-min cadence aliased — games that started and finished between two samples were never observed `running` and got no channel record. At 10 min every running game is sampled 3+ times. The `page[size]=100` (was 20) prevents busy multi-region days from dropping running matches off the captured list (game-specific endpoints have no server-side tier filter, so we over-fetch then filter). This capture feeds the channel↔match binding that lets a completed series resolve its replay during the broadcast. **Note:** this capture cadence is orthogonal to the multi-channel attribution problem — PandaScore's `official`/`main` stream flags can point to a channel that did not actually carry a given regional qualifier (the real broadcast lives on the `official=false` regional channel); resolving that requires trying every Twitch channel in `streams_list` and letting the VOD-window match pick the true broadcaster.
- On drawer open, `fetchMatchStreams(matchIds, startTime, radiantTeam, direTeam)` is called with all sibling game IDs and team names
- `match-streams.js` resolves channels in order: (1) KV fast path `stream:match:{id}`; (2) PandaScore fuzzy match (±2h time window + team name substring matching) - accepts any official Twitch stream (any language), preferring English; (3) ts bucket fallback - returns `_candidates` array of all channels active in that 5-min window
- `findTwitchVod` in `src/api.js` uses the resolved channel exclusively: if `preferredChannel` is set, only that channel is searched on Twitch. `preferredChannel` is set to the definitive `stream:match` KV value or fuzzy-PS-match channel if found; if neither found, falls back to `_candidates[0]` from the ts-bucket IF and only if exactly one candidate exists (multiple candidates means ambiguous concurrent matches — no VOD lookup in that case). If no `preferredChannel` is available after all fallbacks, "No VOD found" is returned.
- `VOD_CHANNEL_LABELS` in `src/api.js` maps known channel handles to display names (ESL, PGL, BLAST, BTS, WePlay, etc.) for the Watch button labels
- `stream:ts:{bucket}` stores a JSON array of all channels active in that 5-min window; `cacheRunningStreams` writes the full array per bucket to avoid last-write-wins collisions
- **Autonomous binding (server-side cron)**: `api/live-matches.js?cron=warm-streams` is triggered every 15 min by a **QStash schedule** (provisioned via `npm run setup-qstash`; see `scripts/setup-qstash-schedules.mjs`), authenticated via `CRON_SECRET`. Moved off GitHub Actions on 2026-06-26 — GHA throttling fired the declared `*/15` cron only every ~1.5-4h in practice, leaving completed series un-bound for hours. `.github/workflows/warm-streams.yml` remains as a manual-dispatch fallback only. It exists because `cacheRunningStreams()` can only write `stream:match` when PandaScore sets `external_identifier` (always null on TI qualifiers and other low-tier events), so a completed series would otherwise get no replay record until a user opened its drawer. The cron pages OpenDota `/promatches` with `less_than_match_id` (up to `WARM_MAX_PAGES`=6 pages, since one 100-match page spans only a few hours on busy days), selects completed tier-1 series within the last 24h via the pure `selectSeriesToWarm()` (groups games by `series_id`, falls back to `match_id`; filters by tier-1 league-name keywords + lookback + presence of both team names), skips any series already fully bound in KV, and drives the existing `/api/match-streams` resolver per series (self-HTTP) so the locked KV→PS-fuzzy→ts-bucket chain writes `stream:match` + Supabase unchanged. **Per-game `starts` param (2026-06-22 fix):** the self-call now includes `starts=id:ts,…` built from a pre-computed `Map` of the already-fetched `odMatches` array, so each sibling row gets its own `started_at` instead of all sharing the series-minimum ts. Writes a run summary to `warm:stream-history:latest` (KV, 8-day TTL). It does NOT modify any locked stream-cache write path. ### Auto-Tweet (Owner Only — NOT Public)
- **This is an owner-only background feature. It must never be exposed in the UI or triggered by users.**
- A GitHub Actions cron (`.github/workflows/auto-tweet.yml`) runs every 30 minutes and POSTs to `/api/draft-posts` with `{ type: "cron" }`, authenticated via `CRON_SECRET`
- `runAutoTweet()` in `api/draft-posts.js` fetches recent pro matches from OpenDota `/api/promatches`, filters to tier-1 leagues (same filter the website uses: `getPremiumLeagueIds()` + `isTier1ByName()` with KV-cached PandaScore names), groups into series, and posts **one tweet per completed series** on X (Twitter)
- Tweet text is a **deterministic template** — no AI involved. Series win: `"{Winner} win {score} over {Loser}\n{Format} | {Tournament}\n{link}"`. BO2 draw: `"{Team1} and {Team2} split the series 1-1\n{Format} | {Tournament}\n{link}"`
- The link always points to the first match of the series via `seriesUrl(games[0])`
- Series tweet includes a 1200x630 OG image (team names + score) generated via `api/og.js?mode=series`
- No `@mentions` — handle tagging was removed because static maps go stale and cannot be reliably verified without manually checking each X account
- Redis (KV) is used to track which series have been tweeted (`auto-tweet:series:{seriesKey}`) to prevent duplicate posts — keys expire after 30 days
- **Concurrent-run lock**: KV key `auto-tweet:lock` (120s TTL, `nx:true`) prevents duplicate tweet runs when GHA and Vercel Cron fire at the same time. Lock is released in `finally` on every exit path.
- `MAX_PER_RUN = 5` caps series tweets per cron execution
- **BO2 draw**: `seriesComplete()` handles the 1-1 draw case explicitly; `makeSeriesTweet` uses the split-series template (`isDraw=true`)
- Series score is always `"{winnerWins}-{loserWins}"` (e.g. "2-0") — both teams initialized to 0 wins before counting
- Twitter auth uses OAuth 1.0a (API Key, API Secret, Access Token, Access Token Secret) — app must have **Read and Write** permissions; tokens must be regenerated after changing permissions
- Required env vars: `TWITTER_API_KEY`, `TWITTER_API_SECRET`, `TWITTER_ACCESS_TOKEN`, `TWITTER_ACCESS_TOKEN_SECRET`, `CRON_SECRET`, `KV_REST_API_URL`, `KV_REST_API_TOKEN`

### Spoiler-Free Mode
- Toggle in `App.jsx` - passed as `spoilerFree` prop to `UpcomingMatches`, `MatchCard`, `DraftDisplay`, and `MatchDrawer`
- **Default is spoiler-free ON for brand-new visitors** (changed June 2026). The `spoilerFree` initializer in `App.jsx` resolves in this order:
  1. `?spoilers=off` URL param → force ON + persist `"true"` (used by the Reddit/X share-link builders in `App.jsx`).
  2. `?spoilers=on` URL param → force OFF + persist `"false"` (symmetric "show scores" share link).
  3. Explicit `localStorage["spoilerFree"]` present → honor it (the user's remembered choice).
  4. No key but `hasPriorFootprint()` true (returning user — any of `theme`, `followedTeams`, `my-teams`, summaries, news-last-visited, calendar-nudge-dismissed, recent-searches, owner present) → OFF (legacy scores-shown), **no nudge**.
  5. Brand-new (no key, no footprint) → ON + first-run nudge eligible.

- **First-run nudge** (`App.jsx`, blue inline-nudge pattern, rendered above the calendar nudge inside the `!initialLoading` block): "Spoiler-free is on" with "Show all scores" / "Keep hidden" actions + X dismiss. Shown **once ever** — dismissal persists to `localStorage["spoiler-nudge-dismissed"]="1"`. Suppressed for url-forced, explicit-pref, and returning-with-footprint users. "Show all scores" sets `spoilerFree=false` + persists.
- **Multi-tab sync**: a `storage` event listener in `App.jsx` updates `spoilerFree` when `localStorage["spoilerFree"]` changes in another tab.
- **Crawler invariant**: spoiler-free is a client-only display preference. `middleware.js` SSR HTML and JSON-LD always contain real scores/winners — spoiler-free must never reach the crawler layer.
- **Analytics**: `spoiler_default_applied {variant: url|explicit|legacy_returning|first_run}`, `spoiler_nudge_shown`, `spoiler_nudge_action {action: show_scores|keep_hidden|dismiss}`, `spoiler_free_toggle {enabled, source: header|settings}`, `spoiler_reveal` (per-match drawer reveal).

All sends route through the batched `dispatchPush` helper (uses `kv.mget`, not per-user gets — folds in pending-refactors #161) which skips users whose per-type toggle is off or who are in quiet hours (`inQuietHours`, evaluated in the subscriber's IANA `tz`; unknown tz → never suppress). Per-type, **series-level** dedup keys (TTL 24h): now-live `push:sent:{seriesId}:{userId}`, starting-soon `push:sent:soon:{seriesId}:{userId}`. Expired subscriptions (HTTP **410 or 404**) are automatically pruned (also deletes `push:prefs`). The service worker collapses repeats via the payload `tag` (`{type}-{seriesId}`). **Analytics funnel (per-type)**: server-side `push_sent` (`{ type, count }`) fires once per `dispatchPush` batch via the GA4 Measurement Protocol (`sendGa4Event`, needs `GA4_API_SECRET`; no-ops without it); client-side `push_opened` (`{ type, matchId }`) fires on the click-through landing. Design/impl plan: `.claude/push-phase1-plan.md`. **Test notification (`POST /api/live-matches?mode=push-test`)**: sends one real web-push round-trip to the subscription posted in the body (no auth — only the requesting browser holds its own endpoint; IP rate-limited to 3/min via `rateLimitByIp`); returns 410 when the push service reports the subscription gone, 502 on other push-service errors. Triggered by the "Send test notification" footer in ManageTeamsModal so users can verify delivery immediately after enabling (the send pipeline is otherwise silent until a followed team actually plays). Requires 4 env vars: `VAPID_PUBLIC_KEY`, `VITE_VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT`. Client utility in `src/utils/push.js` (isPushSupported, getPushPermission, subscribeToPush, `needsIOSInstall` — true when the platform reports push-capable APIs but Apple still blocks delivery outside an installed PWA; see the ManageTeamsModal entry above). `subscribeToPush(teams)` also sends the browser IANA timezone as `prefs.{tz}`; the `push-subscribe` handler **merges** partial prefs over stored ones, so the tz-only auto re-subscribe never clobbers a user's Settings toggles. **Server-derived userId**: The `push-subscribe` POST handler derives `userId = HMAC-SHA256(VAPID_PRIVATE_KEY, subscription.endpoint).slice(0,32)` — the client never sends or controls the userId. The subscription endpoint is unique per browser/device and unforgeable, so no client can overwrite another user's KV keys. **Auto-update on favorites change**: `handleToggleFollow` in `App.jsx` calls `subscribeToPush(next)` automatically whenever `getPushPermission() === 'granted'` -- so following or unfollowing a team immediately re-registers the updated team list with the push server without any extra user action. **Team index cleanup (diff-based)**: The `push-subscribe` POST handler diffs the incoming team list against the previous `push:teams:{userId}` before writing. It removes the userId from `push:team:{name}` for unfollowed teams and adds it for newly followed ones — preventing notifications from stale reverse-index entries. `push:teams:{userId}` is stored as a direct array; a backward-compat read handles pre-fix JSON-stringified entries. **"Sync to your calendar" card** in My Teams is also dismissible with × (persists to `localStorage['calendar-card-dismissed']`).

## Known Issues / Limitations
- **OpenDota indexing lag**: `/promatches` takes 30min to several hours to index games after a series ends. Completed matches only appear once OpenDota indexes them - there is no PandaScore fallback for completed matches.
- Role detection (Carry/Mid/Off/Support) is removed - OpenDota `lane_role` field is unreliable
- VOD channel selection tries three paths: (1) `stream:match:{id}` KV fast path (written while live when `external_identifier` is non-null), (2) PandaScore fuzzy match (±1h window), (3) ts-bucket `_candidates` (used as last resort when exactly one channel was live in that window — prevents false matches in multi-stream time windows). Personal/qualifier streams that PS never links to OD via `external_identifier` can still resolve via path (3).
- Twitch VODs expire after 60 days - old matches will show "No VOD found"
- Search only searches already-loaded matches - user must click "Load more matches" to expand search
- `fetchPremiumLeagueIds` in `src/api.js` fails gracefully (returns empty Set) when OpenDota `/leagues` is down or slow — has an 8-second AbortController timeout so a slow `/leagues` response doesn't delay the full match load
- `fetchProMatches` in `src/api.js` aborts the promatches fetch after 25 seconds (was 15s, increased to tolerate slower OpenDota response times)
- **API resilience**: Error handling is scoped per section. If OpenDota is down, `LatestMatches` shows an inline error with a Retry button while `TournamentHub` and `UpcomingMatches` (PandaScore) continue to render normally. If PandaScore is down, `TournamentHub` and `UpcomingMatches` silently return null (hiding themselves) while `LatestMatches` renders historical OpenDota data normally.
- Live match KV cache must be busted after deploying new fields: `/api/live-matches?bust=1`
- Tournament bracket parsing relies on PandaScore naming conventions ("Round N", "Upper Bracket Semifinal", etc.) for proper round labels. If PandaScore changes naming, rounds fall back to generic "Round N" numbering. Match-named rounds (e.g. "Tundra vs RNX") are detected and shown with no section header.
- PandaScore plan limitation: `GET /dota2/series/{id}` and `GET /dota2/series/{id}/matches` return 404/validation errors on the current plan tier. Use `filter[id]` on `/dota2/series/running|upcoming|past` and `filter[serie_id]` on `/dota2/matches/running|upcoming|past` instead (pattern used in `tournament-detail.js` and `calendar-tournament` mode)
- `findLeague` in `api/_shared.js` uses token overlap to match a series name to an OpenDota league. Collects all candidates with ≥2 token overlap, sorts by overlap descending (ties: non-qualifier before qualifier), then iterates with an inverted numeric guard: if a league has an Arabic numeric token (season number, year) that is NOT in the search's numeric set, it is skipped. **Multi-char Roman numerals in league names (vi, vii, viii, etc.) are normalized to Arabic digits before tokenizing** so "BLAST SLAM VI" contradicts a Season 7 search (6≠7) and "BLAST SLAM VII" matches it (7=7). Single-char Roman numerals (I, V, X) are intentionally excluded from normalization — the token length filter removes them anyway, so "BLAST SLAM I" (OD's single-entry league with no edition number) correctly passes any Arabic season search. OD leagues are cached under `opendota:leagues_v2` (4h TTL, shared by `tournaments.js` and `tournament-heroes.js`). When `findLeague` returns null from cached data, `tournaments.js ?mode=tournament-players` busts the leagues cache and retries once with fresh OD data — handles the case where a new tournament was added to OD after the last cache refresh. After deploying fixes to this function, bust the affected tournament's KV cache with `?bust=1`
- **`name` param construction in TournamentHub**: `tournament.serie` from PandaScore is sometimes just the suffix (e.g. `"Season 29 2026"` without the org prefix `"DreamLeague"`). The frontend combines `tournament.league` + `tournament.serie` into a full name before sending to `tournament-heroes`, using the same pattern as `buildTournamentName`: if `serie` already contains `league`, use `serie` alone; otherwise prepend `league`. This ensures `findLeague` always gets ≥2 token overlap with OpenDota's league names.

## Tier Filtering Strategy

Tournaments are filtered using the tier fields exposed by each data source. No hardcoded name lists.

| Data source | Object type | Tier field | Values accepted |
|---|---|---|---|
| PandaScore `/dota2/matches/*` | match | `match.league.tier` | `'s'`, `'a'` |
| PandaScore `/dota2/tournaments/*` | tournament | `t.tier` (direct, NOT `t.league.tier` which is always null) | `'s'`, `'a'` |
| PandaScore `/dota2/series/*` | series | **no tier field** — always null | derive via `tier1RunningSerieIds` / `tier1UpcomingSerieIds` / `tier1PastSerieIds` sets built from tournament objects |
| OpenDota (leagues/promatches) | league | `league.tier` | `'premium'` (equiv S), `'professional'` (equiv A) |

**PandaScore tier S** = elite international LANs (TI, DreamLeague, ESL One, PGL, BLAST, Riyadh Masters, Premier Series, ...).
**PandaScore tier A** = second-tier professional events (ESL Challenger, regional circuits, ...).
**OpenDota premium** = Valve-sponsored DPC events; equivalent of PandaScore tier S.
**OpenDota professional** = second-tier pro events; equivalent of PandaScore tier A.

Key exports in `api/_shared.js`:
- `isTier1(match)` - checks `match.tournament.tier` (primary field for match objects) then `match.league.tier` for 's' or 'a'
- `isTier1ByName(match, tier1Names)` - **fallback** for when PandaScore hasn't assigned a tier to a newly-created series yet. Checks `match.league.name` against the cached tier1 names array (same list used by `matchesTier1Names`). Used in `live-matches.js` and `upcoming-matches.js` as `isTier1(m) || isTier1ByName(m, names)`. Names are fetched from KV under `KV_TIER1_NAMES_KEY`.
- `KV_TIER1_NAMES_KEY` - exported constant `'dota2:tier1_league_names_v1'` so all consumers read from the same KV key
- `PERMANENT_TIER1_NAMES` - exported hardcoded list of tier1 organizer names (DreamLeague, ESL One, PGL, PGL Wallachia, BLAST, etc.); merged into `names` in `live-matches.js` so tier filtering never depends on KV being warm
- `buildPremiumLeagueIds(leagues)` - pure function; builds a `Set<leagueid>` of OpenDota `premium`-tier leagues only (tier A / professional events are handled via the PandaScore name cache)
- `getPremiumLeagueIds()` - async; fetches `/api/leagues`, caches result in memory
- `setCorsHeaders(req, res, { allowAll })` - sets all CORS response headers; returns `true` on OPTIONS preflight (handler must `return`); `false` otherwise. See CORS Policy section.

Key exports in `src/utils.js`:
- `matchesTier1Names(leagueName, tier1Names)` - returns `true` if leagueName contains any PandaScore tier name, `false` if it doesn't, `null` if the list is empty/all-short. Min name length guard: names < 3 chars are skipped. Called in `fetchProMatches` to filter OpenDota matches; the caller treats only an explicit `true` as a match, so `null` is effectively equivalent to `false` and the OpenDota premium check provides the other branch of the OR.

**GitHub Actions** — `.github/workflows/log-monitor.yml` runs every 2h. Creates a GitHub Issue labeled `monitoring` when critical; comments on existing open `[Alert]` issue to deduplicate. Daily digest at 08:00 UTC only if errors exist that day.

**Cron failure alerts** — `auto-tweet.yml` and `sync-teams.yml` both have `if: failure()` steps that create `[Alert]` issues immediately on cron failure.

**CI function count guard** — `.github/workflows/check-limits.yml` runs on every push to `main` and on every PR. Counts non-underscore `api/*.js` files and fails with an error if the count exceeds 12 (the Vercel Hobby plan limit). Prevents accidental function additions from silently breaking the deploy. Command: `ls api/[^_]*.js | wc -l`.

**Quick check**: `curl -H "Authorization: Bearer $CRON_SECRET" "https://spectateesports.live/api/tournaments?mode=monitor"`

**Stream history reconciliation** — `GET /api/pipeline?type=stream-status` returns the latest reconciliation snapshot (no auth). Run daily at 14:00 UTC as a fire-and-forget tail of the pipeline cron. Compares Supabase `match_stream_history` (permanent) against KV `stream:match:{od_match_id}` (14-day TTL) for the last 14 days. Result persisted under `reconcile:stream-history:latest` (8-day TTL). Fields: `db_rows`, `kv_present`, `kv_expired_or_missing` (expected non-zero near 14-day boundary), `channel_match`, `channel_mismatch` (should always be 0), `ran_at`.

**Supabase tables**:
- `articles` — published editorial articles; written by `_publisher.js` via `getSupabaseAdmin()`; read by `pipeline.js?type=articles` via `getSupabaseAnon()`.
- `match_stream_history` — one row per live game observed by SpectateEsports. Written by `match-streams.js` at two points: (1) **KV fast path** — when `stream:match:{od_match_id}` already exists in KV, a minimal row (`od_match_id`, `channel`, `started_at`, team names) is upserted on every match-page visit; (2) **PS fuzzy match** — a full row including `ps_match_id`, `tournament`, `match_type`, `bracket_round`, `streams_json` is written as soon as PandaScore confirms the match (any stream present), independent of whether a Twitch channel resolves. Both use `ignoreDuplicates: true` so the first write wins. **2026-07-07 fix:** the PS-fuzzy write was previously gated on `getTwitchStreams(...).length > 0` (an official Twitch channel resolving), so a PandaScore-confirmed match with only unofficial/YouTube streams was silently dropped from the table entirely (this is what caused the EWC 2026 archival gap alongside the official-flag issue above). The write is now gated on `normalizeAllStreams(...).length > 0` instead — matching `cacheRunningStreams()`'s condition in `live-matches.js` — with `channel: null` when no official Twitch stream resolves. Channel resolution for the KV fast-path and the caller's `result[id]` (used to embed/link a live channel) is unaffected — still requires a real channel. **Per-game `started_at` (2026-06-21 fix):** `match-streams.js` previously persisted every sibling game of a series with one shared `ts` (the clicked game's start), so games 2/3 inherited game 1's timestamp → game 1's VOD offset. The resolver now accepts an optional `starts=id:ts,…` param; `resolveMatchStreams` (`src/App.jsx`) → `fetchMatchStreams` (`src/api.js`) sends each sibling's own `startTime`, and each row persists its own `started_at` (resolution logic — PS-fuzzy window, ts-bucket — still keys off the primary `ts`). Existing corrupt rows are repaired by `scripts/backfill-vod-offsets.mjs` (`npm run vod-backfill-offsets`): two passes — (1) rows with `ps_match_id` grouped by series, (2) rows with `ps_match_id=null` grouped by `(team_a, team_b, date)` — finds sibling games sharing an identical `started_at`, looks up each game's authoritative OpenDota `start_time`, corrects `started_at`, and nulls `vod_*` (+ clears `match_stream_vods`) so `vod-enrich` re-resolves the correct offset. **2026-06-22:** the warm-streams cron was also producing this corruption (it never passed `starts` to the self-call); fixed in `api/live-matches.js` by building `odStartById` map and passing `starts=id:ts,…` per series. **`live-matches.js` `cacheRunningStreams()` records the broadest data**: as of 2026-06-20 (all-stream-urls) it writes a row for **every** running game with a `begin_at` and ≥1 stream — multi-channel and YouTube-only series included — independent of the LOCKED single-Twitch-channel KV fast-path. `streams_json` now stores **every** stream URL PandaScore returns: all languages, all sources, **official AND unofficial**, each element `{ raw_url, language, official, main, source ('twitch'|'youtube'|'other'), channel }`. `channel` is nullable (YouTube-only series have no Twitch channel). Helper: `normalizeAllStreams()` — exported from `api/_shared.js` and shared by BOTH Supabase write-paths so the persisted shape is identical regardless of which path writes the row first (unit-tested in `__tests__/normalize-all-streams.test.js`). **2026-06-21:** the `match-streams.js` PS-fuzzy path (write point 2 above) previously stored official-only streams via an inline filter; it now uses this shared helper too, so qualifier/fuzzy-resolved rows also capture all languages + unofficial streams (the RU/regional restreams). The single primary `channel` column is still the official Twitch login from `getTwitchStreams()` (the LOCKED VOD anchor — unchanged). Guard-rail: `buildGameUrls()` (`api/pipeline/_vod-urls.js`) primary (`main`) selection now prefers the resolved official channel → official `main` → any official, before any main/first stream, so an unofficial restream never becomes the default replay link (it falls to `others`). Schema: `od_match_id` (PK, unique), `ps_match_id`, `channel` (primary Twitch login, nullable), `started_at`, `team_a`, `team_b`, `tournament`, `match_type`, `game_position`, `bracket_round`, `streams_json` (JSONB), `twitch_vod_id` / `vod_offset_s` / `vod_resolved_at` / `vod_checked_at` / `vod_available` (Phase 1 VOD enrichment fields, populated by `scripts/vod-enrich.mjs`). Backfill scripts in `scripts/`: `backfill-stream-history.mjs` (KV scan → OD lookup), `enrich-stream-history.mjs` (PS fuzzy match), `enrich-stream-history-siblings.mjs` (copy PS data to BO3 G2/G3), `enrich-stream-history-aliases.mjs` (handle OD abbreviations like BB→BetBoom), `backfill-streams-json.mjs` (`npm run backfill-streams-json` — re-fetches PandaScore `streams_list` to widen existing rows' `streams_json` with all-language/unofficial streams; add-only, never shrinks a row; `--days=N` default 14, `--dry-run`; pure decision `computeStreamsBackfill()` unit-tested in `__tests__/backfill-streams-json.test.js`). DDL: `scripts/create-match-stream-history.sql`.
- `match_stream_vods` — per-(game, channel) VOD resolution for **non-main** Twitch channels (alt channels / other languages), so the internal VOD-URL browser can deep-link the start point on those too (P3.3). The main channel's VOD stays in `match_stream_history`. Columns: `od_match_id` + `channel` (composite PK), `language`, `started_at`, `twitch_vod_id`, `vod_offset_s`, `vod_available`, `vod_checked_at`, `vod_resolved_at`. Seeded + resolved by `scripts/vod-enrich.mjs` (alt pass): it derives non-main Twitch channels from `streams_json` (`buildVodSeedRows()`), upserts them (`ignoreDuplicates`), then resolves unresolved rows via the same locked resolver. Read join: `?type=vod-urls` fetches these for the queried matches and `buildGameUrls(row, vodByChannel)` upgrades each matching "other" URL to `kind:'start_point'`. DDL: `scripts/create-match-stream-vods.sql`.

**VOD enrichment** (`scripts/vod-enrich.mjs`, `npm run vod-enrich`): the `vod_*` columns above are null at row-write time and filled by this manual/cron-able job (Phase 1A of `.claude/vod-history-spec.md`). It queries unresolved rows from the last 60 days, calls the **existing locked resolver** `/api/match-streams?mode=twitch-vod&channel=&ts=` (it does **not** reimplement the LOCKED VOD Replay System), and persists the parsed VOD id + offset. Misses on still-live or <24h-old matches are left to retry; misses past the 24h grace window are marked `vod_available = false`. `vod_offset_s` includes the resolver's `+600s` pre-game buffer, so `twitch.tv/videos/{twitch_vod_id}?t={vod_offset_s}s` is the playable link. Two passes: **main channel** → `match_stream_history`, **alt channels** → `match_stream_vods` (P3.3). Pure helpers (`parseVodUrl`, `classifyResolution`, `buildVodSeedRows`) live in `scripts/_vod-enrich-lib.mjs` (unit-tested in `__tests__/vod-enrich-lib.test.js`). Flags: `--dry-run`, `BATCH` env (default 20), `SEED_DAYS` env (default 7, alt-seed window), `VOD_ENRICH_API_BASE` env (default prod). **Automated via QStash** (2026-06-27): `GET /api/pipeline?type=vod-enrich` runs every 15 min, authenticated with `CRON_SECRET`. The handler in `api/pipeline.js` (`handleVodEnrich`) mirrors the script logic — BATCH=10, two passes, same `classifyResolution` helpers. GHA `vod-enrich.yml` remains in place for the PS enrichment piggyback step only. Requires `CRON_SECRET` + `SUPABASE_SERVICE_ROLE_KEY` as Vercel env vars (both already set). **PS enrichment piggyback (2026-06-23):** the same workflow also runs `npm run enrich-history` then `npm run enrich-history-siblings` (`scripts/enrich-stream-history.mjs` + `-siblings.mjs`) to backfill `ps_match_id`/`tournament`/`match_type`/`bracket_round`/`streams_json` for rows born from a PandaScore-blind write path (ts/KV fallback) that never got enriched. A shell gate restricts this to ~once every 3h (the :02 run of hours divisible by 3 UTC, or any `workflow_dispatch`) so PandaScore isn't re-queried for series it will never index; the step also no-ops if `PANDASCORE_TOKEN` is unset. Both scripts honour `ENRICH_LOOKBACK_HOURS` (default 48, invalid values fall back to 48; `0` = full-table one-off backfill); the siblings pass fetches an extra 12h so an edge-of-window null row can still find its enriched sibling. Requires repo secret `PANDASCORE_TOKEN` in addition to the Supabase secrets above.

### Security Headers
`vercel.json` sets four HTTP response headers on all routes via the `headers` config block:
- `X-Frame-Options: DENY` — prevents clickjacking
- `X-Content-Type-Options: nosniff` — prevents MIME-type sniffing
- `Referrer-Policy: strict-origin-when-cross-origin` — limits referrer leakage on cross-origin navigation
- `Permissions-Policy: camera=(), microphone=(), geolocation=()` — disables device APIs the app never uses

### CORS Policy
`setCorsHeaders(req, res, { allowAll })` in `api/_shared.js` centralizes all CORS logic. Sets `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, and `Access-Control-Allow-Headers` on every request; returns `true` and sends `204` on OPTIONS preflight (handler must `return` immediately). `allowAll: true` → `*` (public read endpoints); `allowAll: false` (default) → `https://spectateesports.live` (sensitive endpoints).
- Public (`*`): `live-matches`, `tournaments`, `match-streams` (all modes — `twitch-vod` returns only the VOD URL, never the OAuth token)
- Restricted: `summarize` (Claude API cost), `analytics-chat` (password-gated), `pipeline` (Telegram webhook)

### IP Rate Limiting
`rateLimitByIp(req, kv, limitKey, maxPerMin)` in `api/_shared.js` implements a Redis sliding-window counter keyed by `rl:{limitKey}:{ip}` with a 60-second TTL. Fails open on KV errors. Applied to:
- `/api/summarize` — 10 req/min (protects Claude API spend)
- `/api/analytics-chat` — 10 req/min (protects Claude API spend; applied after auth check)
- `/api/tournaments?mode=watchability` — 20 req/min on cache-miss path only (cached hits bypass the check; protects Twitch Helix quota)
